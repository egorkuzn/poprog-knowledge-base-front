# Техническое задание

## 1. Общие сведения

- **Наименование системы:** Веб-приложение базы знаний (Poprog Knowledge Base Frontend).
- **Условное обозначение:** ИАЭ-ФРОНТ-КБ.
- **Основание для разработки:** потребность в едином пользовательском веб-интерфейсе для доступа к публикациям, студенческим работам, материалам проектов и ИИ-ассистенту.
- **Исполнитель:** команда разработки фронтенда.
- **Заказчик:** владелец портала Poprog.
- **Нормативная база:** структура документа адаптирована по ГОСТ 19.201-78 (Техническое задание), с учетом специфики web/SPA.

## 2. Назначение и цели разработки

### 2.1 Назначение

Система предназначена для предоставления пользователям веб-доступа к материалам портала: публикациям, студенческим работам, проектным направлениям и чату с ИИ-ассистентом.

### 2.2 Цели

- Обеспечить удобную навигацию по разделам портала.
- Реализовать поиск по материалам с переходом по ссылкам из API.
- Обеспечить корректную работу на desktop, compact и mobile режимах.
- Предоставить пользователю возможность взаимодействия с ИИ-ассистентом.

## 3. Характеристика объекта автоматизации

Объект автоматизации: пользовательские сценарии работы с контентом образовательного/научного портала через браузер.

Основные пользовательские роли:

- Гость (неавторизованный пользователь): просмотр контента, поиск, переходы по материалам.
- Пользователь портала: те же сценарии, плюс вход в консоль (через соответствующий интерфейс).

## 4. Требования к системе

### 4.1 Требования к функциональным характеристикам

Система должна обеспечивать:

1. Маршрутизацию и отображение страниц:
   - `/home` (главная),
   - `/publications`,
   - `/works`,
   - `/docs`,
   - `/chat`,
   - `/projects/:itemSlug`,
   - `/not-found`.
2. Общие компоненты интерфейса:
   - верхнюю служебную панель,
   - основное меню навигации,
   - футер.
3. Поиск по порталу:
   - ввод поискового запроса,
   - получение результатов из backend API,
   - отображение карточек результатов.
4. Переход по `link` результата поиска:
   - абсолютные URL (`http/https`) обрабатываются как внешние,
   - относительные URL формируются через `${HOST_API}/{link}`,
   - при отсутствии `link` используется внутренний fallback-маршрут.
5. Проверку перехода на ресурс:
   - для внешних ресурсов показывается предупреждение,
   - для файлов на API-хосте предупреждение не показывается,
   - перед переходом выполняется проверка доступности ресурса,
   - при подтвержденном `404` открывается страница `Ничего не найдено`.
6. Панель проектов:
   - раскрытие по пункту «Проекты»,
   - древовидная навигация в мобильном режиме.
7. Чат с ИИ-ассистентом:
   - отправка сообщений,
   - загрузка истории по `chatId`,
   - сохранение `chatId` в localStorage.
8. Адаптивность интерфейса:
   - корректный режим desktop,
   - compact-режим с агрегированным меню,
   - mobile-режим с мобильным меню.

### 4.2 Требования к надежности

- Система должна корректно обрабатывать ошибки API без падения интерфейса.
- При недоступности истории чата пользователь должен иметь возможность продолжить новый диалог.
- При недоступности внешнего ресурса по проверке `404` пользователь должен быть переведен на `/not-found`.
- Для недетерминированных сетевых ограничений (например, CORS) переход не должен блокировать пользователя.

### 4.3 Требования к условиям эксплуатации

- Поддерживаемая среда: современные браузеры (Chrome, Edge, Firefox, Safari).
- Рекомендуемое минимальное разрешение: 320 px по ширине (mobile).
- Для разработки: Node.js, npm, Vite.
- Для корректной сборки с текущей версией Vite требуется Node.js не ниже `20.19` (или ветка `22.12+`).

### 4.4 Требования к составу и параметрам технических средств

- Клиентская сторона: устройство с браузером и доступом в интернет/локальную сеть.
- Серверная сторона: backend API, доступный по базовому URL (по умолчанию локально `http://localhost:8080`).

### 4.5 Требования к информационной и программной совместимости

- Формат обмена с backend: HTTP/JSON.
- Переменные окружения:
  - `VITE_API_BASE_URL` — переопределение базового адреса API.

### 4.6 Требования к безопасности

- Переход на внешний ресурс должен сопровождаться предупреждением пользователя.
- Для внешних переходов рекомендуется использование отдельной вкладки браузера.
- Не допускается выполнение небезопасных inline-скриптов, кроме служебной страницы-индикатора в контролируемом сценарии проверки перехода.

### 4.7 Требования к эргономике и технической эстетике

- Интерфейс должен быть единообразным на всех основных страницах.
- Навигационные элементы и состояния (hover/active/open) должны быть визуально различимы.
- Ошибочные состояния должны быть понятны пользователю и содержать читаемое сообщение.

## 11. Ограничения текущей версии

1. **Маршрутизация и контент**
   - Маршрут `/projects` в текущей версии отображает главную страницу, отдельной контентной страницы проектов нет.
   - Для неизвестных маршрутов (`*`) выполняется редирект на внешний сайт лаборатории, а не локальная страница 404.
2. **Поиск**
   - Поисковая выдача в интерфейсе запрашивается с лимитом `8` результатов на запрос.
   - Поиск срабатывает с debounce около `250 мс`, мгновенного поиска на каждый символ без задержки нет.
3. **Переходы по ссылкам поиска**
   - Для внешних ресурсов применяется браузерная проверка доступности; при сетевых ограничениях типа CORS точная валидация статуса может быть недоступна.
   - На страницу `/not-found` выполняется переход только при явно подтвержденном `404`.
4. **Адаптивность**
   - Логика режимов и поведения шапки зависит от breakpoints в CSS; ключевой мобильный порог — около `720 px`.
   - В компактных/мобильных режимах часть действий переносится в раскрывающееся меню.
5. **Чат ИИ**
   - История чата привязана к `chatId`, который хранится в localStorage текущего браузера; перенос между устройствами/браузерами не гарантируется.
6. **Интеграционные зависимости**
   - Работа ключевых сценариев зависит от доступности backend API (`/api/search`, `/api/publications/grouped`, `/api/student-works/grouped`, `/api/assistant/*`).
   - При недоступности backend функционал поиска и чата ограничен или недоступен.

## 12. Словарь терминов и сокращений

- **API (Application Programming Interface)** — программный интерфейс взаимодействия frontend и backend.
- **Backend** — серверная часть системы, предоставляющая данные и обработку запросов.
- **Frontend** — клиентская часть системы, отображаемая в браузере пользователя.
- **SPA (Single Page Application)** — одностраничное приложение с клиентской маршрутизацией.
- **Маршрут (Route)** — адрес/путь страницы приложения (например, `/home`, `/chat`).
- **Fallback-маршрут** — резервный маршрут, используемый при отсутствии прямой ссылки.
- **API host / HOST_API** — базовый адрес backend API, используемый для формирования относительных ссылок.
- **Absolute URL** — полный веб-адрес вида `http://...` или `https://...`.
- **Relative URL** — относительный путь без домена, дополняемый базовым адресом.
- **CORS (Cross-Origin Resource Sharing)** — механизм браузерной политики безопасности для междоменных запросов.
- **HTTP 404** — код ответа сервера, означающий, что ресурс не найден.
- **Modal / модальное окно** — всплывающее окно, требующее действия пользователя перед продолжением сценария.
- **Updater (индикатор проверки ресурса)** — промежуточный экран в новой вкладке во время проверки доступности ссылки.
- **Desktop / Compact / Mobile режимы** — варианты адаптивного интерфейса для разных ширин экрана.
- **chatId** — идентификатор диалога в чате ИИ, используемый для загрузки истории сообщений.
